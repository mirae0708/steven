# 💻 직원용 코딩 & 아키텍처 지식 가이드

> Git 저장소에서 직접 추출한 우리 시스템만의 코딩 패턴, 규칙, 아키텍처 원칙 총정리

---

## 🎯 핵심 원칙

### Rule #1: 단일 진입점 (Single Entry Point)
- 하나의 시스템은 하나의 메인 진입점만 허용
- `nexus_orchestrator.py` — 트레이딩 시스템 전체 통합
- `launcher.py` — Invoice Auto 실행기
- `clp_automation_v28_release.py` — CLP 단일 스크립트
- **금지**: 동일 기능의 중복 진입점 생성 (main.py + main2.py + ...)

### Rule #2: 오케스트레이터-모듈 분리
- `Orchestrator`는 흐름 제어만 (데이터 수집→분석→판단→실행)
- `Core Module`은 순수 비즈니스 로직 (지표 계산, 파싱, 저장)
- Orchestrator가 Module 안에 비즈니스 로직을 직접 구현 금지

### Rule #3: Template Method 패턴 (Expert 계층)
```python
# 공통 템플릿: base_expert.py
class BaseExpert:
    def analyze(self, data):
        self.pre_process(data)     # 템플릿
        result = self.do_analyze() # ← 하위 클래스 구현
        self.post_process(result)  # 템플릿
        return result

# 구체 구현: expert_core.py
class QuantitativeExpert(BaseExpert):
    def do_analyze(self):
        return self.calculate_11_indicators()
```

### Rule #4: AI 다수결 의사결정
- 단일 AI 판단 금지 — 최소 3명 분석가 독립 분석 후 다수결
- 각 분석가는 신뢰도(0~1)를 리턴
- 신뢰도 임계값(≥0.5) + 과반수(2/3) 동시 만족 시 실행

### Rule #5: Circuit Breaker 패턴
- 모든 자동화 시스템에 3단계 이상 보호 회로 필수
- 정상 → 주의 → 방어 → 비상 4단계
- 매크로 지표, 기술적 지표, 포트폴리오 리스크 3개 축 검증

### Rule #6: 설정 레이어 분리
- `.env` — API 키, 패스워드 등 시크릿 (Git 추적 금지)
- `config.json` — 환경별 설정 파라미터
- `settings.py` — 코드 기본값 + 환경변수 오버라이드
- **절대**: 코드에 하드코딩 금지

## 🏗️ 아키텍처 패턴 모음

### 패턴 1: 3단계 파이프라인 (Invoice Auto)
```
[Parse] → [Map] → [Execute]
 ↓         ↓          ↓
PDF/AI   고객코드   ERP 입력
데이터   매핑      Double OK
추출     (12회      저장
         재시도)
```
**적용 예**: 모든 ETL 워크플로우, 데이터 수집 파이프라인

### 패턴 2: Watcher 감시→자동처리 (Invoice Auto)
```
Watch 디렉토리 → 파일 감지 → 처리 실행 → 완료 후 재감시 (무한루프)
  - 싱글 인스턴스 (.lock 파일)
  - 예외 발생 시 로그 기록 후 재시작
```
**적용 예**: 파일 모니터링, 메일 수신, API 폴링

### 패턴 3: 단일 스크립트 배포 (CLP Auto)
```python
# 466줄 단일 파일 구성
def load_config()      # 설정 로드
def parse_xxx()        # 처리 함수
def run_automation()   # 메인 로직

if __name__ == "__main__":
    run_automation()
```
**적용 예**: 일반 사용자 배포용, 단순 자동화 작업

### 패턴 4: JS 주입 프론트-백 연결 (Invoice Auto)
```python
# Python에서 JavaScript 실행
from selenium import webdriver
js_code = open("assets/hbl_entry_inject.js").read()
driver.execute_script(js_code, data)
```
**적용 예**: 레거시 웹 시스템 자동화, Selenium DHTML 조작

### 패턴 5: Async Auto-Retry (Invoice Auto CustomerManager)
```python
def ensure_customer(data, attempt=0):
    try:
        return create_or_find(data)
    except DuplicateError:
        if attempt >= 12:
            raise MaxRetryError
        return ensure_customer(data, attempt + 1)
```
**적용 예**: API 호출, DB 트랜잭션, 네트워크 I/O

## 🐍 코딩 규칙

### 1. 파일 구조
```python
"""1줄 요약 설명"""
"""옵션: 상세 설명 (2줄 이상이면 여기)"""

import standard_lib  # 표준 라이브러리
import third_party   # 서드파티 (빈 줄 구분)
import local_module  # 내부 모듈 (빈 줄 구분)

# 전역 상수 (UPPER_CASE)
MAX_RETRY = 12
DEFAULT_MODE = "SEA"

# 설정 클래스
class Config:
    ...

# 비즈니스 로직 함수
def process_invoice():
    ...

# 메인 실행
if __name__ == "__main__":
    ...
```

### 2. 네이밍 컨벤션
| 항목 | 규칙 | 예시 |
|------|------|------|
| 클래스 | PascalCase | `NexusOrchestrator`, `InvoiceParser` |
| 함수 | snake_case | `process_invoice()`, `ensure_customer()` |
| 변수 | snake_case | `pdf_path`, `image_sequence` |
| 상수 | UPPER_SNAKE | `MAX_RETRY = 12`, `DEFAULT_MODE = "SEA"` |
| 비공개 | _prefix | `_snapshot_files()`, `_code_attempt` |
| 모듈 | snake_case | `risk_manager.py`, `hbl_entry.py` |
| 디렉토리 | kebab-case | `99_System_Management/`, `02-Knowledge/` |
| 설정키 | UPPER_SNAKE | `PAPER_TRADING.ENABLED`, `MAX_PORTFOLIO_RISK` |

### 3. 함수 설계 원칙
```python

def calculate_rsi(prices: list[float], period: int = 14) -> float:
    """RSI 계산"""
    ...

# ❌ 나쁜 예 (여러 역할, 부작용 있음)
def process_and_save(data):
    data = clean(data)
    result = calculate(data)
    save_to_db(result)    # ← 부작용! 함수가 2가지 일
```

### 4. 에러 처리 패턴
```python
# ✅ 좋은 예 — 구체적 예외 + 재시도
try:
    result = api_request()
except ConnectionError:
    logger.error(f"연결 실패, 재시도 {attempt+1}/{MAX_RETRY}")
    time.sleep(2 ** attempt)  # 지수 백오프
    return retry(attempt + 1)
except ValueError as e:
    logger.error(f"데이터 형식 오류: {e}")
    raise  # 복구 불가능 → 상위 전파
finally:
    cleanup()

# ❌ 나쁜 예 — 무성 실패
try:
    do_something()
except:
    pass  # ← 어떤 에러인지 모름, 아무것도 안 함
```

### 5. 로깅 패턴
```python
import logging
logger = logging.getLogger(__name__)  # 모듈명 자동

logger.info("처리 시작: %s", pdf_path)       # 정상 흐름
logger.warning("설정 없음, 기본값 사용: %s", key)  # 주의
logger.error("처리 실패: %s — %s", path, e)   # 에러
logger.debug("상세: %s", result)               # 디버깅
# ❌ logger.info("처리 시작: " + pdf_path)  — f-string/format 금지!
```

## 🔒 보안 수칙

1. **API 키 절대 Git 커밋 금지** → `.env` 사용, `.gitignore` 필수
2. **config.json 비밀번호 평문 저장 금지** → 환경변수 참조 `${VAR}`
3. **Selenium 로그인 정보 코드 하드코딩 금지** → `settings.py` or `config.json`
4. **모듈 import 경로 노출 금지** → `__init__.py`로 캡슐화
5. **파일 작업 시 임시파일 정리 필수** → `try/finally` or `contextlib`

## 📋 안티패턴 — 절대 하면 안되는 것

| # | 안티패턴 | 예시 | 이유 |
|:-:|:---------|:-----|:-----|
| ❌ | **하드코딩** | `API_KEY = "sk-xxxx"` | 유출 위험, 변경 어려움 |
| ❌ | **매직 넘버** | `if x > 75:` | 의도 불명, 유지보수 불가 |
| ❌ | **200줄+ 함수** | 200줄 이상 단일 함수 | 읽기 불가, 테스트 불가 |
| ❌ | **무성 catch** | `except: pass` | 버그 은폐, 디버깅 불가 |
| ❌ | **중복 진입점** | `main_old.py`, `main_new.py` | 혼란, 유지보수 지옥 |
| ❌ | **f-string 로깅** | `logger.info(f"...")` | lazy evaluation 손실 |
| ❌ | **전역 mutable** | `global data_list` | 스레드 안전 위반 |
| ❌ | **주석 없는 정규식** | `r'(?P<date>\\d{4}...)'` | 의미 파악 불가 |
